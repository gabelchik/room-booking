# Room Booking Service

Сервис бронирования переговорок.  
Позволяет администраторам управлять переговорками и расписаниями, а сотрудникам бронировать свободные слоты.

---

## Технологии

- **FastAPI** - веб-фреймворк
- **PostgreSQL** - база данных
- **SQLAlchemy 2.0** - ORM
- **Alembic** - миграции
- **APScheduler** - фоновая задача
- **PyJWT** - JWT-токены
- **Docker / Docker Compose** - контейнеризация
- **Pytest** - тестирование

## Запуск приложения

### 1. Клонировать репозиторий
```bash
git clone https://github.com/gabelchik/room-booking.git
cd room-booking
```

### 2. Собрать и запустить
```bash
docker compose up --build
```

- Приложение будет доступно по адресу: localhost:8080
- Swagger документация: localhost:8080/docs

## Тестирование

Запуск всех тестов с отчётом о покрытии.
Для запуска тестов сначала поднимите тестовую базу данных:
```bash
docker compose --profile test up -d db_test
```
Затем выполните тесты:
```bash
docker compose exec app pytest -v --cov=src --cov-report=term
```
После завершения тестов остановите тестовую БД:
```bash
docker compose --profile test stop db_test
```

- **Интеграционный тест** - полный сценарий бронирования комнаты c отменой слота.
- **Юнит-тесты** - генерация слотов, сервисы комнат, бронирований, расписаний, слотов.
- **Покрытие кода** - >70%

## Безопасность

- JWT-токены подписываются секретным ключом, который передаётся через переменную окружения `SECRET_KEY`.
- Доступ к защищённым эндпоинтам контролируется через роль пользователя (`admin`/`user`), извлекаемую из токена.
- Пользователь, пароль, хост, порт базы данных передаются через переменные окружения 
(файл `.env` был специально не добавлен в `.gitgignore`).

| Метод | Эндпоинт                        | Описание            | Доступ      |
| ----- | ------------------------------- | ------------------- | ----------- |

## Принятые решения

### Генерация слотов
Я выбрал предварительную генерацию слотов на период в 8 дней (текущий день + следующие 7) при создании расписания с последующим ежедневным автоматическим добавлением недостающих слотов. 
Это решение опирается на факт, что в 99,9% случаев пользователи интересуются бронированием на ближайшие семь дней, поэтому нет необходимости генерировать слоты на более дальние даты. 
Такой подход обеспечивает высокую скорость получения доступных слотов, поскольку все слоты уже хранятся в базе данных, а запрос сводится к выборке с фильтрацией по дате и статусу брони.
Кроме того, слоты получают стабильные UUID, что необходимо для их идентификации при бронировании, а уникальное ограничение на пару «комната - время начала» защищает от дублирования.
Фоновая задача, запускаемая раз в сутки, поддерживает запас по датам, добавляя слоты на недостающие дни, и делает систему устойчивой к возможным сбоям в расписании генерации.
